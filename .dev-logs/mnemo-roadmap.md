# Mnemo 迭代计划

## 文档目的

这份文档用于记录 Mnemo 后续的产品与技术迭代方向。后续规划、需求讨论、版本拆分时，优先以这份计划作为参考基线；如果后面出现更好的想法，也可以继续在此基础上补充、修正。

## 核心判断

Mnemo 的长期方向，不应该一开始就演化成一个重型 ontology / knowledge graph 系统，而应该沿着这条主线继续升级：

- 先把高质量的语义记忆做稳
- 再逐步补一层轻量结构化记忆
- 最后形成“结构化事实 + 非结构化语义记忆”的双层体系

原因是：

- Mnemo 当前最擅长的是保存用户偏好、项目上下文、决策原因、任务结果、经验教训
- 这些信息里有一部分适合结构化，但也有大量内容天然是“带语境的经验块”
- 如果一开始就强行全面结构化，容易丢掉真实对话中最有价值的上下文

所以 Mnemo 更适合走“语义记忆为主，结构化为辅”的路线。

## 总体路线

### 第一阶段：把现有语义记忆做扎实

目标：让 agent 记得住、找得到、不会越来越乱。

优先方向：

- 改进检索排序质量
- 改进去重与相似记忆合并
- 提高记忆压缩后的质量稳定性
- 增强记忆保存时的质量控制
- 区分短期工作记忆与长期记忆

这一阶段的重点不是扩功能，而是把现有主路径做稳。

### 第二阶段：补轻量结构化层

目标：让最稳定、最明确的事实不漂移、不重复、不相互冲突。

建议只抽三类强结构信息：

- 用户偏好
- 项目事实
- 明确决策

这阶段不做完整知识图谱，不追求大而全，只做最有价值、最稳定的结构化增强。

### 第三阶段：做双层记忆协同

目标：让检索结果既准，又保留必要上下文。

基本思路：

- 检索时同时查结构化事实层和语义记忆层
- 输出时优先给出稳定事实
- 再补充相关背景、原因和历史上下文

这一阶段完成后，Mnemo 会从“记忆笔记工具”升级成“分层长期记忆系统”。

## 具体升级方向

### 1. 语义记忆层继续增强

这是最近几版最优先的方向。

可考虑的能力：

- 更好的检索排序策略
- 记忆重要度加权
- 时间衰减与长期关键记忆保留
- 相似记忆聚合
- 重复内容检测
- 记忆压缩后质量评估
- 搜索结果更稳定地返回真正高价值内容

目标是让 memory_search / memory_get / memory_compress 这一整条链路更可靠。

### 2. 轻量结构化增强

结构化抽取是有必要的，但适合作为增强层，而不是替代层。

建议先使用 typed slots / typed facts 这种轻量方案，而不是完整 ontology 图谱。

例如可抽取的信息形态：

- `user.preference.style = 简洁`
- `project.tech_stack = TypeScript`
- `decision.memory_setup_default_scope = global`

这类结构化信息适合做：

- 去重
- 覆盖更新
- 冲突检测
- 高精度检索

### 3. 作用域模型继续演进

目前 Mnemo 已经有：

- `global`
- `project`

后续可考虑的方向：

- `workspace`
- `client`
- `persona`

但这块不建议过早扩展。短期内应先把 `global / project` 跑顺、跑稳，再考虑更细粒度作用域。

### 4. 记忆质量治理

Mnemo 往后一定会遇到“记忆越来越多、越来越杂”的问题，因此需要逐步加入治理能力。

可以考虑：

- 冲突记忆提示
- 过期记忆提示
- 重复记忆提示
- 建议蒸馏 / 归档 / 删除
- 历史决策的版本化表达

这类能力的价值很高，因为它直接影响长期使用体验。

## 为什么不建议现在就做重型 Ontology

虽然结构化长期记忆很重要，但当前阶段不建议 Mnemo 立即往重型 ontology 系统演化。

原因：

- schema 设计成本高
- 很多高价值记忆天然不是三元组
- 过早图谱化会让系统复杂度快速上升
- 真实收益未必立刻超过当前语义记忆体系

更合适的做法是：

- 保留 Mnemo 当前“蒸馏笔记 + 搜索”的主路径
- 在此基础上补最小结构化层
- 用真实使用反馈决定是否继续往更强 ontology 演进

## 版本规划建议

### 近期待办（优先）

- 持续打磨语义记忆质量
- 继续优化搜索排序与召回稳定性
- 改善去重、合并、压缩策略
- 提高 global / project 两种作用域的使用体验

### 中期方向

- 引入轻量结构化事实层
- 做结构化事实与语义记忆的双路检索
- 增强冲突检测与记忆质量治理

### 远期方向

- 评估是否需要更完整的 ontology / graph 能力
- 评估是否需要更丰富的作用域层级
- 评估是否需要记忆体检、管理面板或调试视图

## 当前产品原则

后续所有迭代，优先遵守以下原则：

- 先增强真实可用性，再扩展概念复杂度
- 先把语义记忆做稳，再加结构化层
- 结构化增强只做最有价值的部分，不追求一步到位
- 不为了“看起来高级”而把 Mnemo 变成过度设计系统
