# 008 - OpenClaw Memory 方案分析与启发

## 背景

阅读了一篇知乎文章（赛博味儿酒咖）对 OpenClaw Memory 实现方案的解读，分析其架构设计对 Mnemo 的启发。

## OpenClaw Memory 架构概述

### 三层记忆分类

OpenClaw 将对话内容分为三类：

1. **会话运行记录**（`~/.openclaw/agents/<agentId>/sessions/*.jsonl`）：运行时自动维护，记录消息、工具调用等，用于会话恢复/压缩/导出
2. **长期记忆**（`MEMORY.md`）：偏好、决策、固定规则等稳定信息，作为 bootstrap 上下文每轮自动注入
3. **日常记忆**（`memory/YYYY-MM-DD.md`）：阶段性记录、当天事项，可追加可回看，默认不自动注入

### 渐进式披露（Progressive Disclosure）

`memory_search` + `memory_get` 两步设计：

- **第一层（索引）**：`memory_search` 返回轻量元数据（标题、日期、类型、Token 消耗量）
- **第二层（详情）**：`memory_get` 按路径与行范围读取原始 Markdown 内容
- **第三层（深潜）**：需要时读取原始源文件

类比人类行为：先扫视标题 → 检阅目录 → 打开文件。极大节省 Token，同时确保精准性。

### Hybrid Search（混合检索）

向量检索 + BM25 全文检索并行：

1. **准备阶段**：会话预热、索引脏检查、触发后台 sync
2. **候选召回**：并行走两条路径——全文检索（精确词项：ID、错误字符串、代码符号）+ 向量检索（语义相近内容）
3. **合并排序**：`mergeHybridResults()` 加权合并（vectorWeight / textWeight），可选 MMR 去重 + 时间衰减，按 minScore 过滤

若 embedding provider 不可用，退化为 FTS-only 模式。

### 向量索引同步策略

- 文件 watcher 监听工作区记忆文件变更
- 脏标记 + 防抖机制（避免写入中被过早读取，合并批量变更）
- 搜索前检查脏位，按需触发后台同步
- 定时兜底同步
- 手动触发索引命令

### Embedding 可配置

支持本地 embedding 和远程 API（OpenAI、Gemini、Voyage、Mistral），可指定 provider 或使用 auto 自动选择，支持 fallback。

### 文件优先理念

- **唯一真源**是 Markdown 文件（MEMORY.md + memory/\*.md），向量库是派生索引，可重建
- Schema free，模型友好，无需 text-to-sql
- 非技术用户友好：直接查看/编辑/迁移文件夹
- 零运维，无供应商锁定

## 与 Mnemo 的对比分析

### Mnemo 已有的优势

| 特性         | 说明                                                                   |
| ------------ | ---------------------------------------------------------------------- |
| LLM 驱动压缩 | `compress` + `compress_apply` 原子操作，OpenClaw 没有自动压缩/蒸馏机制 |
| MCP 标准协议 | 独立 MCP server，可被任何支持 MCP 的 Agent 使用；OpenClaw 记忆是内置的 |
| 零配置       | 不需要配置 embedding provider、数据库后端，开箱即用                    |
| 即时索引     | 所有写入通过 MCP 工具，save 时同步 index，无需文件 watcher             |

### 值得借鉴的方向

#### 1. Hybrid Search —— 高价值，v1.x 建议实现

**问题**：纯向量搜索对精确关键词（错误码、函数名、命令名、配置项）的召回较弱。

**方案**：引入 BM25 风格的全文检索，与向量检索并行，加权合并排序。

**实现思路**：

- 可引入 SQLite FTS5 作为全文检索引擎
- 或者用更轻量的方式：对 note 内容做简单的关键词匹配/TF-IDF 打分
- 搜索时并行走两条路径，合并结果
- 需要权衡复杂度 vs 收益

**收益**：显著提升精确关键词场景的召回质量。

#### 2. 渐进式披露 —— 中等价值，v1.x 可考虑

**问题**：当记忆数量多、单条记忆较长时，`memory_search` 一次性返回完整内容浪费 token。

**方案**：`memory_search` 只返回摘要/首行 + ID + 元数据，需要详情时再读取完整内容。

**当前评估**：Mnemo 的记忆本身就是蒸馏过的精简内容，通常不会很长。当记忆规模增大后（比如上百条）此优化才有实质意义。优先级不高。

#### 3. 三层记忆分类 —— 低优先级

**问题**：是否需要区分长期记忆（自动注入）和日常记忆（按需检索）？

**当前评估**：Mnemo 的 tags 机制已经能覆盖分类需求。Mnemo 的 prompt 指导 Agent 在对话开始时主动搜索相关记忆，效果类似"自动注入"。不必强制引入分类体系。

#### 4. Embedding 可配置 —— 低优先级

**问题**：是否需要支持多种 embedding provider？

**当前评估**：保持零配置、本地优先的定位。MiniLM-L6 对当前规模和使用场景足够。如果未来用户有大量记忆且需要更好的语义理解，可考虑开放配置。

#### 5. 向量索引同步策略 —— 不需要

**原因**：Mnemo 的架构决定了所有记忆写入都通过 MCP 工具，不存在"外部修改文件导致索引不同步"的问题。OpenClaw 需要 watcher 是因为允许 Agent 通过文件工具直接编辑记忆文件，绕过了记忆系统。

## 迭代优先级总结

| 优先级 | 特性                             | 价值   | 版本目标 |
| ------ | -------------------------------- | ------ | -------- |
| ★★★    | Hybrid Search（向量 + 全文检索） | 高     | v1.x     |
| ★★☆    | 渐进式披露（搜索返回摘要）       | 中     | v1.x     |
| ★☆☆    | 三层记忆分类                     | 低     | 暂不实施 |
| ★☆☆    | Embedding 可配置                 | 低     | 暂不实施 |
| —      | 向量索引同步                     | 不需要 | 不实施   |
